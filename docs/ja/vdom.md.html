**The Croquet Virtual DOM Framework** Last Modified: 2021-05-19 # はじめに このフレームワークはネットワーク通信部分にCroquetを使用し、UI部分はウェブ・ブラウザのドキュメント・オブジェクト・モデルに基づいたものとなっている。 ## Croquetのソフトウェア・アーキテクチャ Croquetを使って書かれたアプリケーションは、「モデル」と「ビュー」がはっきりと分離されており、そのことを意識して記述する必要がある。一般的な使用法と同様に、モデルはアプリケーションのロジック部分を指す。ただ、Croquetの場合はより厳密に、「モデルがある状態Sにあり、あるメッセージMを受け取った時、そのメッセージに対する処理を終えた後の状態S'は、そのモデルが*どのようなコンピューター上で実行されているとしても*、SとMから確実に再現でき同じ状態となる」という制約を満たすことを期待されている。 この制約を満たすために、Math.random()にパッチを当てるなどのいくつかの工夫がなされており、一般のプログラマーは、「メッセージハンドラー」として、あるメッセージが来たらそのメッセージに対する処理を自然な形で書くだけで良い。 ビューは、そのようなモデルとユーザーとの仲介をするものである。モデルの状態を表示し、またユーザーがなんらかの入力処理を行った場合には、その入力処理を「同じセッションに参加している全てのモデルに対して」メッセージとして送信することにより、全てのモデルが同じ状態に変換させる。 モデルは同一状態にあることを期待されているが、ビューは各クライアントごとに異なっていても良い。画面サイズ等も違い、またあるユーザーが行なっている操作は、意味がある形でモデルに送信される前にはそのユーザーだけから見えている、ということがあるからである。ビューの動作を記述するコードからは、モデルの状態を読み出すことが許されている。サーバー・クライアントモデルのネットワークプログラミングとのアナロジーを使うと、ビューのプログラムはクライアントプログラムにあたるが、そのプログラムからサーバーの状態を直接読み出すことができるようなものであり、これもまたCroquetの記述性の良さに貢献している。 言い換えると、「クライアント間で共通の情報はモデルに格納し、それ以外はビューに格納する」という原則に従う、ということである。 ## Virtual DOM 上記の基本的なCroquetの仕組みだけでもアプリケーションを作ることは不可能ではないが、ブラウザが提供するドキュメント・オブジェクト・モデル(DOM)を用いてビューを構成するときに、瑣末なコードを繰り返し書くことになり手間がかかる。一言でいいうと、「画面に表示するときに使うDOM要素が保持する情報もまた、大抵の場合は、参加しているクライアント間で共通にしたいことが多い」ので、「ビューが使用するDOMの情報を、仮想的なDOM要素としてモデル内に保持することにより、ユーザー操作に対するハンドラーをモデル内に一度だけ書き、システムが自動的に実際のDOM要素を生成、更新できるようにする」ことによって、コーディングの労力を減らすことができると言える。 この観点に基づいて実装されたのがCroquet Virtual DOM Frameworkである。 ## 動作の記述方法 アプリケーションは単なる状態があるだけではなく、その状態をどのように変化させるかという「動作」の記述も必要である。Croquet Virtual DOM Frameworkでは、「コードもデータとして扱う」ことを基本とすることにより、実行時にユーザーがコードを変更しながらアプリケーションの振る舞いを変えていけるようなオーサリング環境そのものもまた共有化することができるようにもなっている。この共有オーサリング環境は、コード変更の必要がない場合には無効化することもできる。いずれにせよ、コードもまたデータとして扱えることにより、実装の簡略化に寄与している。 以下の例にあるように、このフレームワークでは動作をある特定の規約に従って書く。文法としては"class"というキーワードを流用しているが、実際にはtraitsやexpandersと呼ばれる言語機能に触発された、クラス拡張機能となっている。簡単に言えば、仮想DOM要素に対して、「要素そのものを変更することなく、要素の外部に付随したものとして新たな振る舞いを追加し、call()という特別な文法で呼び出すことができるメソッドの組」である。 ## 概念・用語 ### モデル モデルとは上記のように、アプリケーションのロジック部分を書くために使われるオブジェクトである。外部からのメッセージをうけとることによって状態変化を起こす。モデルという言葉は、アプリケーションのロジック全体を指す場合もあるが、`Model`として説明されている場合には、Croquet.Modelクラスを継承したクラスの個々のインスタンスを現すこともある。言い換えれば、アプリケーションのロジック部分は、複数の`Model`クラスから作られており、それらのインスタンスが複数使われていることが普通である。 ### ビュー ビューはモデルの状態を画面に表示し、またユーザーからの入力を受け付けてモデルにメッセージとして送信する。`Model`と同様に、`View`と言った場合には`Croquet.View`を継承したクラスのインスタンスで、アプリケーション内に複数あっても良い。 ### クライアント 基本的にはブラウザーで動作するCroquetアプリケーションである。クライアントはあるCroquetセッションに参加する。そのセッションに参加しているクライアントは同じ状態にあるモデルを共有する。 ### リフレクター ネットワーク上にあり、ビューから送られてきたメッセージを、そのセッションに参加している全てのクライアントに配送する役割を担うサーバー。 ### 仮想DOM要素 Croquet Virtual DOM Frameworkでは、実際のDOMのインターフェイスを模倣したオブジェクト(`Element`と呼ばれるクラスのインスタンス)がモデル内に作られる。複数のクライアント間で同一であるべき部分は、この仮想DOM要素のプロパティーを書き換えることによって実装されることになる。 ### 仮想DOMのビュー `Element`のインスタンスが画面に表示される際には`ElementView`というViewオブジェクトが生成される。`ElementView`は実際のDOM要素を保持しており、対応する`Element`のプロパティーが変更された場合に、差分を実際のDOM要素に反映させる役割を担う。また、実際のDOM要素でイベントが発火した場合などには、それをCroquetのメッセージとしてモデルに送信する。 ### 実際のDOM要素 仮想DOMビューが保持しており、ブラウザーが実際に画面に表示するもの。 ## 仮想DOM要素への参照 他の仮想DOM要素への参照(ポインタ)は`ElementRef`というデータ構造として表現されている。`ElementRef`は、ビューとモデルの分離に付随するずれを吸収するための簡単な仕組みである。 例えば、複数のユーザーが同時に操作しているグラフィカルな共有アプリケーションがあり、そこである一人のユーザーがある要素を画面上で動かそうとしたとしよう。このような場合には、要素を移動させるというメッセージがリフレクターを経由してモデルに届くまでの間に、その要素はすでに他のユーザーによって削除されている可能性がある。そのため、モデル側では、関与する要素を常に検索し、それがまだ存在することを確認する必要がある。`ElementRef`はこのような検索ステップを統一的に行う仕組みを提供する。 ### expander `Element`または`ElementView`に「インストール」できる動作を、メソッドの集合として記述したもの。ある特定の規約に従って書かれたJavaScriptのクラスである。 # 例題1 (Counter) 以下ではまずは簡単な例題を紹介しよう。 ~~~~~~~~ JavaScript init() { ~~~~~~~~ `init()は、expanderがある要素に「インストール」されるときに自動的に実行される特別なメソッド名である。(ただし、オーサリングツール内でexpanderが編集されることがあり、その場合は編集のたびにinit()が呼び出される。) ~~~~~~~~ JavaScript メソッド内の`this`は、expanderではなくそれがインストールされたDOM要素を指す。 ~~~~~~~~ JavaScript if (this._get("count") === undefined) { this._set("count", 0); } ~~~~~~~~ 上記のように、`init()`は既に存在している要素に対して複数回呼ばれることがある。そのため、`init()`内で変数を初期化をする時、そして同じ要素が生き残っている間はコードが少々変わっても値を保持しておいてほしい場合は、このように値がまだセットされていないときにのみ初期化をするというスタイルとなる。ちなみに、`addEventListener`は複数回呼んでも一度だけ登録されるようになっている。ここでは`count`というプロパティーが仮想DOM要素に定義される。 `_get()`と`_set()`は仮想DOM要素のプロパティを読み書きするためのメソッドである。仮想DOM要素はJSON化可能な値だけを保持することができるが、単純にドット表記で読み書きできるようにしてしまうと、ついつい自由変数を持った関数などをしまってしまうため、「JSON化できるオブジェクトだけが格納できる」ということを覚えておき、さらに内部的に使用しているプロパティー名と衝突しないようにするために、敢えて`_get()`と`_set()`という冗長な表記にしている。 ~~~~~~~~ JavaScript this.future(1000).call("Counter", "next"); ~~~~~~~~ `future()`は`setTimeout()`のようなものである。引数には論理時間でのミリ秒を指定する。reflectorが送ってくるheartbeatを元に、参加している全てのクライアントが指定された時間が経過した後で、その後に指定されているメソッドを起動する。通常は、この論理時間は実時間と同期されているので、実時間でも約1000ミリ秒である。 この場合に指定されているのは`call()`というメソッドであり、「`Counter`という名前でインストールされているexpanderの`next`というメソッドを起動する」という意味である。近い将来、他のexpanderメソッドの呼び出しと同様に、`this.future(1000).next()`とexpanderの名前を書かなくても良くなり、その場合には「同じexpander」が暗黙的に指定されている」こととなる予定である。 ~~~~~~~~ JavaScript next() { let c = this._get("count") + 1; this._set("count", c); this.value = "" + c; this.future(1000).call("Counter", "next"); } ~~~~~~~~ 最後の行は、`next()`から`next()`自身を再び呼び出す。`requestAnimationFrame()`またはsetTimeout同様これは一定時間ごとに処理をする際の定石である。 ~~~~~~~~ reset() { let c = 0; this._set("count", c); this.value = "" + c; } ~~~~~~~~ 新しいアプリケーションの開発とデバッグ 現時点では、フレームワーク部分のパッケージ化はあえて避け、ファイルを書き換えてページをリロードすればすぐに試せるという形で開発を進めている。新しいアプリケーションを書く際には、上記のcounter.jsのようなファイルを新しく作り、またそれを呼び出すcounter.htmlのコピーを作れば良いだろう。 ## `isLocal`フラグ 開発の多くの時間は、見た目の調整や、その他モデル動作の確認をするために、共有部分を試すのではなく、一台のクライアントのみを使えばよい作業に費やされることが多い。そのような開発サイクルを簡略化するために、URLに`?isLocal`というクエリー・パラメーターが指定されている場合には、リフレクターの動作をエミュレートした形で実行される。counter.htmlの例であれば、 ~~~~~~~~ http://localhost:8000/counter.html?isLocal ~~~~~~~~ とすると、全く同一のコードが、ネットワークを必要とすることなく動作する。 ## debugger 現時点では、expanderとして書いたコードは一度文字列表現を経由してから評価され実行されるため、ブラウザのデバッガが行番号を表示できないという問題がある。内部に`debugger`文を書くことによりデバッガーを起動することができ、またconsole.log`からの出力があれば、コンソールに出力された結果の行番号をクリックすることにより"VM...."という擬似ファイル名のコードを参照してブレイク・ポイントを挿入することもできる。将来的には、可能な時には文字列を経由せずにexpanderとして挿入されることを許すなどの改良が望ましい。 # 例題2 (Drawing) 次にクライアントごとにビューが異なる状態を持つ例として、簡単なお絵描きプログラムの例を紹介する。こちらは、一つのライブラリーファイルに複数のexpanderを書き、ビューにもコードを追加する例である。 ~~~~~~~~ JavaScript class DrawModel { init() { this._set("color", "black"); this.subscribe(this.id, "line", "line"); this.subscribe(this.sessionId, "color", "color"); this.subscribe(this.sessionId, "clear", "clear"); this.subscribe(this.id, "pointerUp", "savePersistentData"); } ~~~~~~~~
